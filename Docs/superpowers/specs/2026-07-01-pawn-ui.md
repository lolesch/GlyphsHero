# Pawn UI (status bar + selected-pawn HUD) — Design Spec

**Date:** 2026-07-01
**Status:** Approved (grilled with Leonid 2026-07-01) — issues may be filed.
**Scope:** Two new pawn-facing UI surfaces (a grid-following status bar and a selected-pawn HUD),
a new click-**selection** concept, and the small supporting changes they need. Presentation +
selection input only — no combat/chain/economy rules change. Two-way door (UI), so no ADR; decisions
are logged in the [Slice ledger](#slice-ledger).

---

## Problem

There is no in-game readout of a pawn's health/mana, and no selection concept at all:

- `HexSelectionHandler` (`Assets/Code/Runtime/Core/HexSelectionHandler.cs`) only fires
  `OnPawnHovered`. The pawn inventory (`PawnInventoryView`) therefore updates on **hover**, which is
  wrong — it should update on **click selection**.
- `PawnResourceView` (`Assets/Code/Runtime/UI/PawnResourceView.cs`) exists but is single-resource,
  never paints an initial value, and never unsubscribes (`OnCurrentChanged` leak).
- `PawnStats` already models `health`/`mana` as `Resource` plus `healthRegen`/`manaRegen`/
  `movementSpeed`/`range`, but nothing surfaces them to the player.

## Goal

Two surfaces, both keyed off a single new **selection** concept:

> **A grid status bar follows every pawn showing health + mana. A HUD shows the *selected* pawn's
> full readout. Selection is by click; hover only highlights the effect shape.**

---

## Decisions

### 1. Selection model (new)
- Selection lives on `HexSelectionHandler`: add `OnPawnSelected` (event) + `SelectedPawn` (property),
  matching the existing static-event style of `OnPawnHovered`.
- **Click** a pawn (left mouse down over its hex) selects it; clicking another switches.
  **No deselect** for now — clicking empty does nothing, re-clicking the same pawn is not a toggle.
- Any pawn is selectable, **including enemies**.
- Hover keeps ONLY the effect-shape tilemap highlight. **Q/E rotation moves to the *selected* pawn.**
  - *Known tension (logged):* Q/E rotates the selected pawn while the *hovered* pawn's shape is the
    one highlighted — they can visually mismatch. Left as-is for now.

### 2. Grid status bar
- All pawns (both teams), **always visible**.
- World-space canvas **child on `Pawn.prefab`** — follows automatically, every spawned pawn gets it.
- Two fill bars (health top, mana bottom), **no text**.
- New `PawnStatusBar` (UI asmdef) self-binds from the parent `Pawn.Stats` in `Start()` (runs after
  `PawnFactory`'s synchronous `SpawnPawn`, so `Stats` is ready). Respects **UI → Pawns** layering —
  `Pawn`/`PawnFactory` must not reference UI.

### 3. Selected-pawn HUD
- **Separate** panel; shares only the selection source (`OnPawnSelected`). May be unified with the
  inventory later — kept separate now for modularity.
- Shows: icon + name; health + mana bars **with numeric current/max text**; secondary stats
  (`healthRegen`, `manaRegen`, `movementSpeed`, `range`).
- **Fully live**: pools via `Resource.OnCurrentChanged`, every stat (incl. max) via
  `Stat.OnTotalChanged`. Unsubscribe the previous pawn on selection switch and on destroy.
- Hidden until the first selection. Any pawn (incl. enemies) shows.

### 4. Supporting Statistics changes (pure, testable)
- Add a public forwarding event to `Stat`:
  `event Action<float> OnTotalChanged` passing through to `MaxValue`. Do **not** expose `MaxValue`
  itself — `MutableFloat` has public mutators, so handing out the object would bypass `Stat` as the
  sole modifier gate. Value reads stay on the implicit `float` operator.
- Guard `Resource.Percentage` against `MaxValue <= 0` (a 0-mana pawn currently yields `0/0 = NaN`).

### 5. Reuse / references
- `ItemTooltipController` is the reference for a lifecycle-managed, ref-wired presentation panel.
- `PawnResourceView` is reused (fixed) for both grid bars; the HUD bars add numeric text.

---

## Out of scope (cross-referenced, not folded in)
- **Enemy inventory should be read-only / non-interactable** while player inventories stay draggable —
  a separate `InventoryView`/drag-interactability concern, not part of these issues.
- Unifying the HUD and inventory into one "inspect pawn" surface — a later pass.
- Resolved-attacks summary in the HUD — deferred (overlaps the item tooltip's job).

## Implementation note (night runner)
The runner writes **C# only** and a **WIRE IN UNITY** recipe; a human authors the prefab child, the
HUD panel, and all SerializeField wiring, then verifies in Rider/Unity. Only the Statistics issue has
a clean red-green seam. Issues (dependency order): Statistics → Selection → Inventory-trigger →
Grid-bar → HUD.

## Slice ledger
- **Assumptions (two-way doors, review/veto):**
  - Status bar is a world-space prefab child (vs a screen-space projected manager).
  - Grid bar shows bars only; exact numbers live in the HUD.
  - HUD placement/colors are left to the human authoring the panel (tuning).
- **Decisions taken:**
  - No deselect for now (click-only, switch-on-click). — cheap to revisit.
  - `Stat` gets a *forwarding event*, not an exposed `MaxValue` — encapsulation (Stat stays the
    modifier gate). — door test: keeps the one-way "who can mutate stats" invariant intact.
  - HUD kept separate from inventory (modularity now, unify later).
- **Gaps left open (one-way doors — NOT filled here):**
  - Q/E-selected vs hover-highlighted mismatch: noted, not redesigned.
  - Enemy-inventory read-only: separate issue, not defined here.
